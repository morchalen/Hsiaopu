# 10 实习面试题库与回答模板

## 项目介绍题

**Q：介绍一下你的项目。**

A：Hsiaopu 是一个 Android AI 助手 demo，使用 Kotlin、Jetpack Compose 和 MVVM 思路开发。它支持 AI 聊天、流式回复、会话管理、多 Provider 配置、Shell 命令执行和 Shizuku 系统控制。UI 层用 Compose，业务状态集中在 `ChatViewModel` 的 `StateFlow`，本地会话用 Room 保存，设置用 DataStore 保存，网络层通过 `AiProvider` 接口抽象 DeepSeek 和 OpenAI 兼容接口。

**Q：你负责或重点理解了哪些模块？**

A：重点理解了四块：第一是 Compose 页面和响应式布局；第二是 ViewModel 中发送消息、流式响应和工具调用流程；第三是 Room/DataStore/Hilt 这套本地数据和依赖管理；第四是 Shizuku + Shell 的系统命令执行。

## Android 题

**Q：Activity 在项目中做什么？**

A：`MainActivity` 是单 Activity 入口，负责安装 Splash、开启 edge-to-edge、通过 `setContent` 加载 Compose 根组件，并通过 `@AndroidEntryPoint` 接入 Hilt。真正页面由 Composable 函数组成。

**Q：Compose 和 XML 的区别？**

A：XML 是命令式更新控件，Compose 是声明式 UI，把状态映射为界面。这个项目聊天页状态变化频繁，比如消息列表、loading、流式内容，用 Compose + StateFlow 可以让 UI 自动重组，代码更集中。

**Q：ViewModel 有什么作用？**

A：ViewModel 保存和管理 UI 相关业务状态，配置变化时比 Activity 稳定。本项目的会话、消息、加载状态、错误、token 统计都在 `ChatViewModel` 中，通过 StateFlow 暴露给 UI。

## Kotlin 题

**Q：StateFlow 和 Flow 区别？**

A：Flow 是数据流，只有 collect 时才执行，可以连续发射值；StateFlow 是状态流，始终有当前值，适合 UI 状态。本项目 DataStore 和 Room 返回 Flow，ViewModel 对外暴露 StateFlow。

**Q：协程在项目里怎么用？**

A：ViewModel 用 `viewModelScope.launch` 启动异步任务，数据库和网络不阻塞主线程；系统命令和 IO 操作用 `Dispatchers.IO`；AI 流式响应用 Flow 持续 emit chunk。

**Q：data class 的好处？**

A：自动生成 `copy`、`equals`、`toString` 等，适合表达不可变状态。项目的 `ChatUiState` 用 data class，每次通过 copy 更新字段。

## Java/JVM 题

**Q：项目里哪里体现了多态？**

A：`AiProvider` 是接口，`DeepSeekProvider` 和 `OpenAICompatibleProvider` 是实现。ViewModel 通过 Registry 调用 Provider，不依赖具体实现，新增模型服务时扩展实现类即可。

**Q：注解处理是什么？**

A：注解是元数据，Hilt 和 Room 在编译期根据注解生成代码，比如 DAO 实现、依赖注入组件。这样减少手写样板，也能在编译期检查部分错误。

**Q：反射有什么风险？**

A：反射绕过编译期检查，运行时才发现错误，也可能受库版本变化影响。本项目 `ShizukuHelper` 反射调用 `newProcess`，功能可行，但正式项目要关注兼容性。

## 数据与架构题

**Q：Room 表怎么设计？**

A：两张表：`conversations` 保存会话，`messages` 保存消息。messages 通过 `conversationId` 外键关联 conversations，并设置级联删除，删除会话时消息一起删除。

**Q：为什么设置不用 Room？**

A：API Key、模型名、主题属于简单键值配置，用 DataStore 更轻量；Room 更适合结构化关系数据，比如会话和消息。

**Q：Hilt 注入了什么？**

A：Application 启动 Hilt，Activity 使用 Hilt ViewModel，ViewModel 构造注入 Repository、ProviderRegistry、SettingsDataStore 和 ApplicationContext。DatabaseModule 提供 AppDatabase 和 DAO。

## 网络题

**Q：流式回复怎么实现？**

A：Retrofit 接口使用 `@Streaming` 返回 ResponseBody，Provider 按行读取 SSE，处理 `data:` 开头的 chunk，解析 JSON 中的 delta content，然后通过 Flow emit 给 ViewModel。ViewModel 拼接到 `streamingContent`，Compose 观察状态实时刷新。

**Q：Provider 抽象有什么好处？**

A：ViewModel 不需要关心具体 AI 服务，只通过统一接口发送消息。DeepSeek 和 OpenAI Compatible 都实现同一接口，后续扩展新 Provider 时改动小。

## 系统权限题

**Q：Shizuku 是什么？**

A：Shizuku 是一个让普通应用在用户授权后调用高权限能力的框架。本项目用它执行 shell 命令和系统控制，比如读取 dumpsys、设置亮度、控制 WiFi 等。

**Q：Shell 命令失败怎么处理？**

A：先检查 Shizuku 是否可用和授权，不满足就返回错误结果。执行异常会被捕获，包装成 `ShellResult` 或 `SysResult`，UI 显示错误而不是崩溃。

## 反问和优化题

**Q：你觉得项目哪里还能优化？**

A：第一，`ChatViewModel` 职责偏多，可以拆分 Chat、Settings、SystemTool ViewModel 或 UseCase。第二，Provider 重复代码可抽基类。第三，工具调用用正则解析不够稳，可改结构化 JSON。第四，危险系统操作需要代码层二次确认。第五，API Key 应加密保存，日志也要避免打印敏感信息。
